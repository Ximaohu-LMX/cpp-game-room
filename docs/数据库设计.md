# 数据库设计

本项目采用吃鸡式单胜者对战背景：每个房间开启一局 battle，一局游戏只有一个最终胜者，其他参与玩家均为失败方。当前代码实现的是简化战斗流程，用移动、攻击、血量和存活判断模拟战斗，不实现完整商业游戏中的地图、装备、子弹、技能、安全区、回放等复杂玩法。

## MySQL 与 Redis 分工

项目同时使用 MySQL 和 Redis。

MySQL 负责最终持久化数据：

- 玩家基础数据
- 玩家最终积分
- 玩家累计胜负
- 战斗总记录
- 每个玩家在一局战斗中的结果
- 结算日志和幂等控制

Redis 负责实时排行榜视图：

- 使用 Sorted Set 保存玩家积分排名
- 支持快速查询排行榜 TopN
- 作为 MySQL 中 `player.score` 的查询加速视图

整体关系是：

```text
MySQL 是最终数据来源
Redis 是排行榜缓存和读加速
```

战斗结算时，服务端先通过 MySQL 更新玩家积分和结算日志，再把玩家最新积分写入 Redis 排行榜。Redis 数据如果丢失，可以通过 MySQL 的 `player` 表重建。

## MySQL 表设计

### player

玩家表，保存玩家基础数据和累计战绩。

| 字段 | 说明 |
| --- | --- |
| player_id | 玩家 ID，主键 |
| name | 玩家名 |
| score | 积分，默认 1000 |
| win_count | 胜场数 |
| lose_count | 败场数 |
| create_time | 创建时间 |
| update_time | 更新时间 |

### battle

战斗总表，一场 battle 一条记录。

| 字段 | 说明 |
| --- | --- |
| battle_id | 战斗 ID，主键 |
| room_id | 房间 ID |
| winner_id | 本局唯一胜者 |
| start_time | 开始时间，当前预留 |
| end_time | 结束时间，当前预留 |
| result_json | 战斗结果扩展字段 |

### battle_player_result

战斗玩家结果表，一场 battle 中每个参与玩家一条记录。

| 字段 | 说明 |
| --- | --- |
| result_id | 自增主键 |
| battle_id | 战斗 ID |
| player_id | 玩家 ID |
| result | 本局结果，WIN 或 LOSE |
| score_delta | 本局积分变化 |

唯一键：

```sql
UNIQUE KEY uniq_battle_player_result (battle_id, player_id)
```

该唯一键保证同一场战斗中，同一个玩家只有一条结果记录。

### settlement_log

结算日志表，用于保证积分结算幂等。

| 字段 | 说明 |
| --- | --- |
| settlement_id | 结算日志 ID |
| battle_id | 战斗 ID |
| player_id | 玩家 ID |
| score_delta | 积分变化 |
| created_at | 创建时间 |

唯一键：

```sql
UNIQUE KEY uniq_battle_player (battle_id, player_id)
```

该唯一键保证同一场战斗中，同一个玩家只能被结算一次。

## Redis 数据结构设计

当前 Redis 只用于排行榜。

### rank:score

排行榜 key：

```text
rank:score
```

数据类型：

```text
Sorted Set
```

成员和值：

```text
member = player_id
score  = 玩家当前积分
```

示例：

```redis
ZADD rank:score 1020 1001
ZADD rank:score 980 1002
ZADD rank:score 1100 1003
```

查询排行榜时使用：

```redis
ZREVRANGE rank:score offset stop WITHSCORES
```

`ZREVRANGE` 会按积分从高到低返回玩家列表。

当前 Redis 不保存：

- battle 记录
- 玩家每帧输入
- 战斗状态快照
- 结算日志
- 玩家完整资料

如果 Redis 中的排行榜数据丢失，可以从 MySQL 重建：

```text
SELECT player_id, score FROM player;
-> ZADD rank:score score player_id
```

## 未落库的战斗内数据

当前数据库只持久化玩家基础数据、战斗最终结果、每个玩家的胜负结果、积分结算日志和排行榜数据。

以下战斗内过程数据没有落库：

- 玩家每一帧输入
- 每一帧状态快照
- 移动轨迹
- 攻击过程明细
- 伤害明细
- 死亡时间线
- 战斗回放数据

这些数据目前只在服务端内存中参与 tick 推进。流程是：

```text
InputRequest
-> InputBuffer
-> GameLoop 50ms tick 消费输入
-> GameRoom 更新位置、血量、存活状态
-> StateSync 广播状态快照
```

因此，当前版本能跑通“匹配 -> 房间 -> 简化战斗 -> 胜负判断 -> 结算 -> 排行榜”的主链路，但不支持基于数据库的战斗回放、操作审计或完整反作弊分析。

## room_id 与 battle_id

`room_id` 表示房间 ID，是玩家匹配后进入的房间实例。

`battle_id` 表示战斗 ID，是一局具体对战和结算的唯一标识。

当前项目中，一个 room 只打一局 battle，打完后房间关闭，所以两者看起来是一一对应的。但概念上不合并，方便后续扩展：

- 一个房间连续打多局
- 房间重开
- 观战房间
- 自定义房间

## 一场战斗的数据库操作流程

假设一局 4 人：

```text
room_id = 10
battle_id = 1000
winner = 1
losers = 2, 3, 4
```

游戏结束时，`SettlementService::SettleBattle` 执行以下流程。

### 1. 生成 battle_id

```cpp
const int64_t battle_id = IdGenerator::NextBattleId();
```

### 2. 插入 battle 总记录

```text
battle_id | room_id | winner_id
1000      | 10      | 1
```

这条记录表示：这是一局战斗，胜者是玩家 1。

### 3. 插入胜者的玩家结果

```text
battle_id | player_id | result | score_delta
1000      | 1         | WIN    | +20
```

### 4. 插入失败玩家的结果

```text
battle_id | player_id | result | score_delta
1000      | 2         | LOSE   | -10
1000      | 3         | LOSE   | -10
1000      | 4         | LOSE   | -10
```

### 5. 插入胜者的 settlement_log

```text
battle_id | player_id | score_delta
1000      | 1         | +20
```

如果插入成功，说明玩家 1 尚未结算过，于是更新玩家数据：

```text
player.score += 20
player.win_count += 1
```

随后把玩家 1 的最新积分写入 Redis 排行榜：

```redis
ZADD rank:score new_score 1
```

### 6. 插入失败玩家的 settlement_log

每个失败玩家插入一条结算日志：

```text
battle_id | player_id | score_delta
1000      | 2         | -10
1000      | 3         | -10
1000      | 4         | -10
```

插入成功后更新玩家数据：

```text
player.score -= 10
player.lose_count += 1
```

随后把失败玩家的最新积分写入 Redis 排行榜：

```redis
ZADD rank:score new_score 2
ZADD rank:score new_score 3
ZADD rank:score new_score 4
```

### 7. 查询排行榜

客户端请求排行榜时，服务端从 Redis 查询：

```redis
ZREVRANGE rank:score offset stop WITHSCORES
```

返回结果按积分从高到低排列。

## 为什么需要 settlement_log

战斗结算可能被重复触发，例如：

- 服务重试
- 网络异常后补偿
- 进程崩溃恢复
- 重复收到结算请求

如果没有幂等控制，玩家可能被重复加分或扣分。

因此 `settlement_log` 使用 `(battle_id, player_id)` 唯一键保证：

```text
同一场 battle，同一个 player，只能结算一次
```

如果插入结算日志失败，就跳过后续积分更新。

## 总结

```text
battle 记录一局游戏整体结果
battle_player_result 记录每个玩家在本局中的胜负和积分变化
settlement_log 保证积分更新幂等
player 保存最终玩家积分和累计胜负
Redis ZSET 保存排行榜实时视图
```
